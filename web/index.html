But why?

diskwhy scans your drive and finds what's actually using space — not just "big file on Desktop", but node_modules, Docker images, Xcode caches, stale Python bytecode. Then removes them safely.

how it works

Three commands.

No config required. Scan to see what's there. Review with --dry-run. Remove what you approved.

01
Scan
diskwhy scan

Walks your home directory and catalogs space hogs by category. Add --deep to include system dirs like /usr/local and Homebrew. Add --path /dir to target a specific directory.

02
Review
diskwhy clean --all --dry-run

Preview exactly what would be deleted — size, category, and staleness for each item. Items modified in the last 7 days are marked active and always skipped.

03
Clean
diskwhy clean --all --yes

Remove what you approved. Pass --trash to move to Trash instead of permanent delete. Git repos run git gc — the repo stays intact, loose objects are pruned.

categories

Knows what to look for.

Most disk scanners show you file sizes. diskwhy knows what node_modules is, what Xcode DerivedData is, and why your ~/.npm cache is 4 GB after six months of inactivity.

node_modules
npm, yarn, pnpm dependency trees
git_objects
Loose objects and pack files in .git — runs git gc, not raw delete
docker
Dangling images and unused volumes via Docker API
pycache
Python __pycache__ directories and .pyc bytecode files
pip_cache
pip download and wheel cache (~/.cache/pip)
npm_cache
npm global cache in ~/.npm
brew_cache
Homebrew downloads cache (macOS)
xcode_derived
Xcode DerivedData and build artifacts (macOS)
apt_cache
apt package cache in /var/cache/apt (Linux)
snap_cache
Snap package cache (Linux)
logs
Log files matching *.log and *.log.* patterns
trash
Items in system Trash / ~/.Trash
staleness scores

Every item gets a staleness score from access time, mtime, or sentinel files (package.json, go.mod, Cargo.toml). Scores are frozen at scan time — the clean phase never re-derives them.

active modified < 7 days ago — never deleted
recent accessed < 30 days ago
stale not accessed in 30–90 days
unused not accessed in > 90 days
features

Built the right way.

No shell-outs. No brittle heuristics. No silent data loss. Engineered to run safely on real developer machines.

Active items are never deleted

Items modified in the last 7 days are marked active and unconditionally skipped by the clean phase. Cache categories (pycache, brew, npm, pip, etc.) are always safe regardless of staleness — they regenerate on demand.

Staleness frozen at scan time

Access time, mtime, and sentinel files (package.json, go.mod, Cargo.toml) are evaluated once during the scan. The clean command never re-reads them — what you saw in the preview is exactly what gets acted on.

Docker API, not exec.Command

Docker integration uses the github.com/docker/docker SDK directly — no exec.Command("docker", "images"). Queries containers, images, and volumes from the daemon socket. Silently skips if Docker is unavailable.

JSON output, schema versioned

Every command supports --json with a stable schema_version: 1 contract. Pipe to jq, wire into dashboards, or integrate with your own tooling. See the schema changelog.

Git repos run gc, not delete

The git_objects category runs git gc --prune=now instead of nuking .git. Loose objects are pruned and pack files are consolidated. The repo stays intact. The disk space comes back.

Interactive shell built in

Run diskwhy with no arguments to open a REPL. Type /scan, /clean --all --dry-run, /help. Adaptive to dark and light terminals. No readline dependency.

Start in 30 seconds.

Requires Go 1.21+. Works on macOS and Linux.
